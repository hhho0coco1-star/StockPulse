| 항목 | 값 |

| JWT 발급 | `POST /api/v1/auth/login` → `data.accessToken` |

## 시나리오별 결과

### 시나리오 A — 시세 조회 (`GET /api/v1/market/quote/{symbol}`)

| 항목 | 목표 | 실측값 | 판정 |
|------|------|--------|------|
| 최대 RPS | 1000 | 추정치* | 미측정 |
| p95 응답시간 | < 300ms | 추정치* | 미측정 |
| p99 응답시간 | — | 추정치* | 미측정 |
| error_rate | < 1% | 추정치* | 미측정 |

> \* Redis 캐시 hit 경로 기준 로컬 단일 인스턴스 추정: p95 < 50ms, 500 RPS 달성 가능. 1000 RPS는 Redis 병렬 처리 및 JVM warm-up 이후 측정 필요.

**실행 명령**:
```bash
k6 run tests/load/scenarioA_quote.js
k6 run --summary-export=tests/load/out/A.json tests/load/scenarioA_quote.js
```

### 시나리오 B — 모의투자 주문 (`POST /api/v1/orders`)

| 항목 | 목표 | 실측값 | 판정 |
|------|------|--------|------|
| 최대 RPS | ~80 | 추정치* | 미측정 |
| p95 응답시간 | < 500ms | 추정치* | 미측정 |
| error_rate | < 1% | 추정치* | 미측정 |
| business_rejected (잔고부족) | — | 별도 집계 | — |

> \* Saga 비동기 경로 (202 Accepted) — 동기 응답만 측정. Kafka publish까지 포함 시 Kafka latency 추가.

**실행 전 준비**:
```bash
curl -X POST http://localhost:8080/api/v1/account/reset \
     -H "Authorization: Bearer $K6_JWT_TOKEN"
```

**실행 명령**:
```bash
k6 run tests/load/scenarioB_order.js
```

| 항목 | 목표 | 실측값 | 판정 |
|------|------|--------|------|
| 최대 RPS | ~30 | 추정치* | 미측정 |

```bash

```

## 병목 분석 (추정 — 실측 후 갱신)

| 구간 | 예상 병목 | 개선 방향 |
|------|----------|---------|
| Gateway → market-collector | WebFlux 버퍼 / Redis 커넥션 | Redis 커넥션 풀 증대 |
| Trading Saga | Kafka publish latency | Kafka acks=1, batch.size 조정 |
| Discussion POST | PostgreSQL HikariCP pool | pool-size 20으로 증대 |
| 전체 | JVM Cold start | GraalVM Native or warm-up 요청 |

> Grafana → StockPulse Overview → p95 latency 패널로 실측 후 이 섹션 업데이트.
> Tempo → 느린 Trace 추적 → 어느 span에서 지연 발생하는지 확인.
