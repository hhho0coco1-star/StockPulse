# StockPulse 검증 — 경계면 교차 비교 + 실행 증거

검증의 핵심은 "파일이 있는가"가 아니라 **"경계면에서 데이터 모양이 일치하는가"**, 그리고 **"실제로 빌드·테스트가 통과하는가"**다. 추정이 아니라 실행 결과로 판정한다.

## 왜 경계면인가
MSA 버그의 대부분은 서비스 안이 아니라 서비스 사이에서 난다. producer가 보내는 이벤트와 consumer가 받는 타입이 한 글자만 달라도 런타임에 깨진다. 양쪽을 동시에 열어 대조해야 잡힌다.

## 워크플로우

### 1. 검증 대상 수집
- backend-dev가 알린 구현 파일 목록과 `_workspace/{wbs}_architect_spec.md`를 읽는다
- 스펙의 DoD를 검증 항목으로 전환한다

### 2. 경계면 교차 비교 (핵심)
각 경계면에서 **두 쪽 파일을 동시에 열어** shape을 대조한다:

| 경계면 | 비교 대상 A | 비교 대상 B | 확인 |
|--------|------------|------------|------|
| REST | 컨트롤러 응답 DTO | 소비하는 클라이언트/프론트 기대 타입 | 필드명·타입·nullable 일치 |
| Kafka | producer 발행 이벤트 | consumer 역직렬화 타입 | 스키마·필드·직렬화 방식 일치 |
| DB | 마이그레이션/스키마 | JPA 엔티티 | 컬럼명·타입·제약·인덱스 일치 |

불일치는 **양쪽 파일의 줄 번호와 함께** 구체적으로 기록한다.

### 3. 실제 실행
빌드·테스트를 돌리고 그 출력을 증거로 삼는다 (Gradle 예시):
- 빌드: `./gradlew :{module}:build`
- 테스트: `./gradlew :{module}:test`
- 통합(Testcontainers) 포함 시 컨테이너 기동 가능 여부 확인

실행 불가 환경이면 PASS로 추정하지 말고 **"실행 불가"로 명시**한다.

### 4. 판정·보고
`_workspace/{wbs}_qa_report.md`에 기록:
```markdown
# QA 리포트: {WBS} {작업명}
## 판정: PASS / FAIL
## 경계면 비교
- {경계면}: 일치 / 불일치(파일:줄, 내용)
## 실행 증거
- 명령: ... / 결과: ...
## 발견 이슈 (줄 번호 포함)
## DoD 충족
- [ ] 항목별 충족 여부
```
- FAIL이면 backend-dev에 재작업 요청(최대 2~3회), 설계-구현 불일치면 architect에 보고

## 작업 원칙
- **실행 증거 없이 "통과" 주장 금지.** 명령과 출력을 근거로 댄다
- 전체 완성 후 1회가 아니라, 모듈/작업 완료 직후 **점진적으로** 검증한다
- 한쪽만 보고 판단하지 않는다 — 반드시 경계면 양쪽을 대조
- 문제는 줄 번호와 함께 지적한다

- [ ] 빌드·테스트를 실제 실행하고 출력을 기록
