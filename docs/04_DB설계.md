# StockPulse DB 설계

> MSA 원칙: **각 서비스가 자신의 데이터를 소유**(DB per service)하고, 다른 서비스 데이터는 API/이벤트로만 접근한다. 03 API가 요구하는 데이터에서 도출. 토픽은 [02_시스템아키텍처.md](02_시스템아키텍처.md) 참고.

---

## 1. 저장소별 역할

| 저장소 | 용도 | 소유 서비스 |
|--------|------|------------|
| **PostgreSQL** | 트랜잭션 데이터(회원·주문·잔고·보유·커뮤니티·알림) + 각 서비스 Outbox | Auth/User, Account, Portfolio, Trading, Community, Notification |
| **TimescaleDB** | 시계열(틱·캔들) — 대량 시간 기반 데이터 | Market Collector |
| **MongoDB** | 비정형 문서(뉴스·인사이트 카드) | News Collector, Insight |
| **Redis** | 시세 캐시·랭킹·Rate Limit·Pub/Sub·세션 | Market, Ranking, Gateway, Realtime, Auth |

> 데이터 정합성: 서비스 경계를 넘는 상태 변경은 **Outbox + Kafka 이벤트 + Saga**로 처리(즉시 일관성 대신 최종 일관성).

## 2. PostgreSQL 테이블 (서비스별)

### 2.1 Auth/User
**users**: `id(PK) · email(unique) · password_hash · nickname · provider(LOCAL/GOOGLE/KAKAO) · created_at`
**watchlist**: `id(PK) · user_id(FK) · symbol · created_at` · **unique(user_id, symbol)**

> Refresh 토큰·세션은 Redis(`refresh:{userId}`)에서 관리.

### 2.2 Account (가상 잔고)
**account**: `user_id(PK) · cash(예수금) · reserved(주문 예약금) · updated_at`
**ledger**(원장): `id(PK) · user_id · order_id · type(RESERVE/CONFIRM/RELEASE/SETTLE) · amount · balance_after · created_at`
**outbox**: `id(PK) · aggregate_type · aggregate_id · event_type · payload(jsonb) · status(PENDING/SENT) · created_at`

### 2.3 Portfolio (보유)
**holdings**: `id(PK) · user_id · symbol · quantity · avg_buy_price · updated_at` · **unique(user_id, symbol)**
**outbox**: (Account와 동일 구조)

### 2.4 Trading (주문·Saga)
**orders**: `id(PK) · user_id · symbol · side(BUY/SELL) · type(MARKET/LIMIT) · quantity · price · status(RECEIVED/RESERVED/EXECUTED/COMPLETED/REJECTED/COMPENSATING/CANCELLED) · filled_price · created_at`
**order_saga**: `order_id(PK) · current_step · step_status · compensation_reason · updated_at` — Saga 상태 머신
**outbox**: (동일 구조)
**processed_event**(멱등성): `event_id(PK) · processed_at` — 중복 소비 방지

### 2.5 Community
**posts**: `id(PK) · symbol · user_id · content · like_count · comment_count · created_at`
**comments**: `id(PK) · post_id(FK) · user_id · content · created_at`
**post_likes**: `post_id · user_id · created_at` · **PK(post_id, user_id)**
**outbox**: (동일 구조)

### 2.6 Notification/Alert
**alert_rules**: `id(PK) · user_id · symbol · type(TARGET_PRICE/CHANGE_RATE/NEWS) · condition(jsonb) · enabled · created_at`
**device_tokens**: `id(PK) · user_id · token(unique) · platform(ios/android/web) · created_at`
**notifications**(이력): `id(PK) · user_id · type · title · body · data(jsonb) · read · created_at`

> **공통 Outbox 패턴**: 상태 변경 서비스(Account·Portfolio·Trading·Community)는 비즈니스 테이블과 `outbox`를 **같은 로컬 트랜잭션**으로 기록 → 별도 발행기가 Kafka로 전송 → Kafka 장애에도 이벤트 무유실.

## 3. TimescaleDB (시계열)

**ticks** (하이퍼테이블): `time(ts) · symbol · price · volume` — 파티션 키 `time`, 인덱스 `(symbol, time desc)`
**candles** (연속 집계/Continuous Aggregate): `bucket(ts) · symbol · interval(1m/5m/1d) · open · high · low · close · volume`

> 원시 틱은 보존기간 정책(retention)으로 자동 정리, 캔들은 연속 집계로 자동 롤업. 차트 조회는 candles에서.

---

## 4. MongoDB 컬렉션

**news**: `{ _id, symbol, title, url, source, publishedAt, sentiment(POSITIVE/NEUTRAL/NEGATIVE), keywords:[], score }`
**insights**(전망 카드): 
```json
{
  "_id": "005930",
  "symbol": "005930",

  "news": { "score": 75, "positive": 3, "negative": 0, "topNewsIds": [] },
  "updatedAt": "2026-06-19T..."
}
```

## 5. Redis 키 설계

| 키 | 타입 | 용도 | TTL |
|----|------|------|-----|
| `quote:{symbol}` | Hash | 최신 시세(price·change·volume) 캐시 | 갱신형 |
| `candle:recent:{symbol}:{interval}` | List/ZSet | 최근 윈도우(차트 초기 로드용) | 단기 |
| `ranking:{period}` | Sorted Set | 수익률 랭킹 (member=userId, score=수익률) | 갱신형 |
| `ratelimit:{ip}` | String(counter) | Gateway Rate Limit(분당 N건) | 60s |
| `refresh:{userId}` | String | Refresh 토큰 | 토큰 만료 |
| `pubsub: market:{symbol}` | Pub/Sub | 시세 fan-out 채널 | - |
| `pubsub: chat:{symbol}` | Pub/Sub | 채팅 fan-out 채널 | - |
| `pubsub: insight:{symbol}` | Pub/Sub | 인사이트 갱신 fan-out 채널 | - |

> Realtime Gateway 다중 인스턴스는 `pubsub:*` 채널을 구독해 어느 인스턴스에 연결된 클라이언트든 동일 메시지를 받는다(수평 확장·무중단).
