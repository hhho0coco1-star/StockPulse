# StockPulse 시퀀스 다이어그램

> 주요 플로우의 시간순 상호작용. mermaid `sequenceDiagram`으로 표현(GitHub·VS Code에서 렌더). 읽는 법은 문서 하단 참고.

---

## 1. 로그인 + 인증된 요청 (Gateway 중앙 인증)

```mermaid
sequenceDiagram
    participant C as Client
    participant G as API Gateway
    participant A as Auth Service
    participant DB as PostgreSQL
    participant R as Redis
    participant P as Portfolio

    C->>G: POST /auth/login (email, pw)
    G->>A: 라우팅
    A->>DB: 사용자 조회·비밀번호 검증
    A->>R: Refresh 토큰 저장 (refresh:{userId})
    A-->>G: Access + Refresh 토큰
    G-->>C: 토큰 반환

    Note over C,G: 이후 모든 요청은 Bearer 토큰 첨부
    C->>G: GET /portfolio (Authorization: Bearer)
    G->>G: JWT 검증
    G->>P: X-User-Id 헤더로 전달
    Note right of G: 각 서비스는 JWT 직접 검증 안 함
    P-->>C: 보유·수익률 데이터
```

## 2. 실시간 시세 푸시

```mermaid
sequenceDiagram
    participant KIS as KIS API
    participant MC as Market Collector
    participant K as Kafka
    participant TS as TimescaleDB
    participant RT as Realtime Gateway
    participant RD as Redis Pub/Sub
    participant C as Client(WS)

    KIS-->>MC: 실시간 틱 (WebSocket)
    MC->>K: market.tick 발행
    MC->>TS: 틱 적재 (캔들 연속집계)
    K->>RT: market.tick 소비
    RT->>RD: publish market:{symbol}
    RD-->>RT: 모든 RT 인스턴스가 수신
    RT-->>C: /topic/market/{symbol} 푸시
    Note over RT,C: 다중 인스턴스라도 Redis Pub/Sub로<br/>동일 메시지 fan-out
```

## 3. 모의투자 Saga (성공 + 보상)

```mermaid
sequenceDiagram
    participant C as Client
    participant T as Trading
    participant K as Kafka
    participant A as Account
    participant P as Portfolio
    participant Rk as Ranking

    C->>T: POST /orders (매수)
    T->>T: order(RECEIVED) + outbox 저장
    T-->>C: 202 Accepted (orderId)
    T->>K: order.events: ReserveBalance

    K->>A: 소비
    alt 잔고 충분
        A->>A: 잔고 예약 + outbox
        A->>K: account.events: Reserved(OK)
        K->>T: 소비 → 체결(EXECUTED)
        T->>K: order.events: UpdateHoldings
        K->>P: 보유 갱신 + outbox
        P->>K: portfolio.updated
        K->>T: 소비 → ConfirmOrder(COMPLETED)
        K->>Rk: 평가·랭킹 갱신
        T-->>C: WS /user/queue/orders: 체결 완료
    else 잔고 부족
        A->>K: account.events: Rejected
        K->>T: 소비 → 주문 거절(REJECTED)
        T-->>C: WS: 주문 거절
    end

    Note over T,P: 보유 갱신(P) 실패 시 → 보상:<br/>ReleaseBalance(A) + 주문 COMPENSATING→CANCELLED.<br/>모든 소비는 멱등(processed_event)
```

## 4. 인사이트 4축 산출 + 백테스트

```mermaid
sequenceDiagram
    participant MC as Market Collector
    participant NC as News Collector
    participant FC as Fundamentals Collector
    participant K as Kafka
    participant I as Insight (Kafka Streams)
    participant TS as TimescaleDB
    participant M as MongoDB
    participant RT as Realtime Gateway

    MC->>K: market.tick
    NC->>K: news.raw
    FC->>K: fundamentals.raw
    K->>I: 3개 스트림 소비·조인
    I->>I: 모멘텀(윈도우 집계) · 실적 · 뉴스 감성 점수
    I->>TS: 과거 PER 밴드 조회
    I->>I: 밸류에이션(업종평균·밴드 대비 상대 위치)
    I->>M: 4축 종합 전망 카드 저장
    I->>K: insight.updated
    K->>RT: 소비 → 화면 갱신 푸시

    Note over I,M: [별도 주기 작업] 백테스트<br/>과거 점수 vs 이후 수익률 → 적중률 집계 → backtests 저장
```

## 읽는 법 (시퀀스 다이어그램이란)

- **세로 = 시간**: 위에서 아래로 시간이 흐른다.
- **세로 점선(생명선)**: 각 참여자(서비스)가 살아있는 동안.
- **화살표**:
  - 실선 `->>` = 요청/메시지 보냄
  - 점선 `-->>` = 응답/비동기 전달
- **`alt / else`**: 조건 분기(성공/실패 등 경우의 수).
- **`Note`**: 부연 설명.

> 시퀀스 다이어그램은 "**여러 서비스가 시간 순서로 어떻게 주고받는지**"를 한눈에 보여준다. 특히 3번 Saga처럼 여러 서비스가 이벤트로 얽힌 흐름을 이해·검증할 때 가장 유용하다(어디서 실패하면 어떻게 보상하는지가 명확해짐).
