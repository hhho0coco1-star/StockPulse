# StockPulse 스펙 작성 — WBS를 구현 계약으로

WBS 작업 한 개를 backend-dev가 추가 질문 없이 구현할 수 있는 스펙으로 만든다. 스펙의 품질이 구현 품질을 결정하므로, 추측이 아니라 `docs/` 설계 문서를 근거로 작성한다.

## 왜 스펙을 먼저 쓰는가
코드를 바로 짜면 서비스 간 계약이 제각각이 되어 MSA 경계에서 버그가 터진다. 스펙 단계에서 계약(REST 응답 모양, Kafka 이벤트 스키마, DB 테이블)을 먼저 못 박으면, 구현·검증이 같은 기준을 공유하게 된다.

## 워크플로우

### 1. 입력 확인
- 오케스트레이터가 지정한 WBS 작업 번호 확인 (예: `0-1 Gradle 멀티모듈 골격`)
- `docs/05_WBS_개발계획.md`에서 해당 작업의 범위·의존·DoD를 읽는다

### 2. 근거 문서 수집
대상 작업에 관련된 문서만 선택적으로 읽는다 (전부 읽지 않는다):
| 작업 성격 | 우선 읽을 문서 |
|----------|---------------|
| 서비스 신규/구조 | `02_시스템아키텍처.md` |
| API 추가/변경 | `03_API설계.md` |
| 데이터 저장 | `04_DB설계.md` |
| 비즈니스 흐름 | `08_시퀀스다이어그램.md` |
| 기술 선택 근거 | `07_기술선택이유.md` |

문서에 근거가 없으면 **"문서 누락"으로 명시**하고 합리적 제안을 덧붙인다. 임의 확정하지 않는다.

### 3. 스펙 작성
`_workspace/{wbs}_architect_spec.md`에 아래 구조로 쓴다:

```markdown
# 스펙: {WBS 번호} {작업명}

## 대상
- 서비스/모듈: {예: services/auth-service, common}
- 의존 작업: {선행 WBS 번호 또는 없음}

## 인터페이스 계약
### REST (해당 시)
- METHOD /path → 요청 DTO(필드:타입), 응답 DTO(필드:타입), 에러 응답(코드/형식)
### Kafka (해당 시)
- 토픽명, 이벤트 스키마(필드:타입), producer/consumer 서비스
### DB (해당 시)
- 테이블/컬렉션, 컬럼(타입/제약), 인덱스, Outbox 필요 여부

## 데이터 모델
- 도메인 객체와 필드

## 완료 기준 (DoD)
- [ ] 검증 가능한 항목 (qa가 PASS/FAIL 판정할 수 있게)

## 참고
- 근거 문서 경로 + 미해결/문서 누락 항목
```

### 4. 구현 후 문서 동기화 (구현 완료 시 호출됨)
- 실제 코드와 `docs/` 해당 문서를 대조해 불일치를 찾는다
- 코드가 옳으면 문서를 갱신, 설계가 옳으면 불일치를 backend-dev에 보고
- 변경한 문서와 요약을 리더에게 보고

## 작성 원칙
- backend-dev가 "이건 어떻게?"라고 되묻지 않을 만큼 구체적으로 (필드명·타입·에러까지)
- 한 번에 WBS 작업 1개만. 범위를 넓히지 않는다
- 초보 개발자가 읽으므로 생소한 용어(Outbox, Saga 등)에는 한 줄 설명을 붙인다
- 설계 문서 간 모순은 임의 해결 금지 — 양쪽 출처를 병기해 리더에게 결정 요청

## 체크리스트
- [ ] WBS 작업 1개로 범위 한정
- [ ] 관련 docs를 근거로 인용 (추측 아님)
- [ ] REST/Kafka/DB 계약 중 해당하는 것 모두 명시
- [ ] DoD가 qa가 판정 가능한 형태로 작성됨
- [ ] 문서 누락/모순은 명시적으로 표시
