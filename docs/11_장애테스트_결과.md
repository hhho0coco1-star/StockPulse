## 시나리오별 관찰 절차

### 시나리오 1 — Pod kill

```bash
kubectl apply -n stockpulse -f tests/chaos/pod-kill.yaml
kubectl get pods -n stockpulse -l app.kubernetes.io/name=trading-service -w
```

**기대 관찰**:
- Grafana `up` 패널: trading-service 0 → 1 복귀
- k6 error_rate 일시 상승 후 회복 (<5% 목표)
- MTTR = Pod Terminating → Running(Ready) 시간 (초)

---

### 시나리오 2 — Redis 지연 (CB 전이)

```bash
kubectl apply -n stockpulse -f tests/chaos/network-delay.yaml
watch -n 10 "curl -s http://localhost:8082/actuator/circuitbreakers | jq '.circuitBreakers.redis.state'"
```

**기대 관찰**:
- Actuator state: `CLOSED` → `OPEN` (요청 200ms 초과 → TimeLimiter 발동)
- CB OPEN 중: `GET /api/v1/market/quote/{symbol}` 응답 503, `error.code=UPSTREAM_UNAVAILABLE`
- fallback 경로 로그: `TickRepository.findLatestBySymbol` 호출 확인
- 90s 후 격리 해제 → `HALF_OPEN` → `CLOSED` 복귀
- Grafana → CB state 패널에서 전이 그래프 확인

---

### 시나리오 3 — Kafka 격리 (DLT)

```bash
kubectl apply -n stockpulse -f tests/chaos/kafka-partition-offline.yaml
# 60s 후 자동 복구
kafka-console-consumer \
  --bootstrap-server localhost:9092 \
  --topic market.tick.DLT \
  --from-beginning --max-messages 10
```

**기대 관찰**:
- 격리 중 서비스 로그: `KafkaProducerException` + DLT 라우팅 메시지
- 복구 후 DLT 토픽에 메시지 적재 확인
- consumer lag Grafana 패널에서 0 수렴 확인
