## Phase 1 — 시세 파이프라인

### [2026-06-26] KIS WebSocket 연결 실패 (`Invalid handshake response getStatus: 200 OK`)

- **증상**: `market-collector` 기동 시 KIS WebSocket 연결이 `Invalid handshake response getStatus: 200 OK`로 계속 실패.
- **원인**: `application.yml`의 `kis.ws-url`이 `wss://openapivts.koreainvestment.com:29443/websocket`(KIS 개발자 포털 예시 URL, 이미 닫힌 접속)으로 잘못 설정되어 있었음. 해당 URL은 HTTP 200 응답(HTML 페이지)을 반환해 WebSocket 핸드셰이크가 거절됨.
- **해결**: GitHub 공개 KIS API 구현체 소스 분석으로 실제 운영 URL `ws://ops.koreainvestment.com:31000` 확인 후 수정.
- **배운 점**: KIS 공식 포털이 접근 불가 상태일 때 공개 구현체 코드를 레퍼런스로 활용하는 것이 유효. WebSocket 400 대신 200이 나오면 "HTTP 서버가 WS 핸드셰이크를 이해 못함"으로 진단.

### [2026-06-26] `parseTick`이 모든 틱 데이터를 무시

- **증상**: KIS WebSocket 연결 성공 후 틱이 Kafka로 발행되지 않음. 아무 로그도 없음.
- **원인**: `KisWebSocketClient.kt:84` — `if (raw.startsWith("{") || raw.startsWith("0|")) return`. KIS 틱 데이터 포맷이 정확히 `0|{종목코드}|...`로 시작하는데, `"0|"` 조건으로 모든 틱을 얼리 리턴.
- **해결**: `if (raw.startsWith("{")) return` 으로 수정 (JSON 제어 메시지만 스킵).
- **배운 점**: 외부 API 데이터 포맷을 실제 패킷으로 먼저 확인 후 파싱 조건 작성. 추가 로그(`log.info("틱 수신: $symbol")`) 없이는 무음 버그를 발견하기 어려움.

### [2026-06-26] TimescaleDB JPA `column "id" does not exist`

- **증상**: Spring Boot 기동 시 `ERROR: column "id" does not exist` — JPA가 `ticks` 테이블에서 `id` 컬럼을 찾으려 함.
- **원인**: TimescaleDB hypertable(`ticks`)은 시계열 특성상 serial `id` PK 없이 `(time, symbol)`만 있음. `@Entity + JpaRepository`는 `@Id`가 있는 컬럼을 필요로 해서 런타임 오류 발생.
- **해결**: `Tick.kt`에서 `@Entity` 제거, 단순 `data class`로 변경. `TickRepository`를 `JpaRepository` → `JdbcTemplate` 직접 SQL로 교체(`INSERT INTO ticks (time, symbol, price, volume) VALUES (?, ?, ?, ?)`).
- **배운 점**: TimescaleDB hypertable은 JPA와 직접 호환 안 됨. 시계열 저장소는 처음부터 JdbcTemplate/R2DBC 직접 SQL을 사용.

## Phase 4 — 커뮤니티·실용기능

### [2026-06-27] discussion-service Redis 인증 실패 (`NOAUTH HELLO must be called with the client already authenticated`)

- **증상**: discussion-service 기동 시 Redis 연결 실패. `NOAUTH HELLO must be called with the client already authenticated`.
- **원인**: `docker-compose.yml`의 Redis에 `requirepass changeme`가 설정되어 있으나, `discussion-service/application.yml`에 `spring.data.redis.password` 항목이 누락됨.
- **해결**: `spring.data.redis.password: ${REDIS_PASSWORD:changeme}` 추가.
- **배운 점**: 서비스별 yml 작성 시 공통 인프라(Redis/Kafka) 인증 설정을 체크리스트로 관리할 것. `NOAUTH` 에러는 즉시 인증 미설정을 의심.

## Phase 5 — 안정성

### [2026-06-27] Resilience4j AOP가 suspend 함수에 미적용

- **증상**: `KisApprovalClient`의 suspend 함수에 `@CircuitBreaker` 어노테이션 적용 후 CB가 동작하지 않음.
- **원인**: Resilience4j AOP 프록시는 JVM 코루틴 suspend 함수를 인식하지 못함. suspend 함수는 컴파일 시 `Continuation` 파라미터가 추가되어 AOP 포인트컷이 매칭 실패.
- **해결**: AOP 대신 코루틴 decorator 방식 직접 적용 (`circuitBreakerRegistry.circuitBreaker("kis").executeCoroutine { ... }`).
- **배운 점**: Spring Boot + Kotlin Coroutine 환경에서 Resilience4j AOP는 suspend 함수에 작동하지 않음. 코루틴 전용 decorator API를 사용해야 함.

### [2026-06-27] TradingService 자기 호출로 CB/Retry 미동작

- **증상**: `TradingService` 내부에서 `@CircuitBreaker` 어노테이션이 붙은 같은 클래스 메서드를 호출해도 CB가 열리지 않음.
- **원인**: Spring AOP 프록시 자기 호출 문제 — 같은 빈 내부 메서드 호출은 프록시를 우회함.
- **해결**: Kafka 발행 로직을 별도 `OrderEventPublisher` 컴포넌트로 분리. `TradingService → OrderEventPublisher`로 위임.
- **배운 점**: AOP 기반 CB/Retry 어노테이션은 반드시 별도 빈에 적용해야 함. 같은 클래스 내 호출은 AOP가 작동하지 않음.

## Phase 6 — 관측

### [2026-06-28] OTLP 트레이스 내보내기 실패 (`unexpected end of stream on http://localhost:4317`)

- **증상**: 서비스 기동 후 로그에 `HttpExporter - Failed to export spans: unexpected end of stream on http://localhost:4317` 반복. Tempo에 트레이스가 수신되지 않음.
- **원인**: architect 스펙이 `endpoint: http://localhost:4317`(gRPC 포트)로 작성했으나, Spring Boot 3.3의 `micrometer-tracing-bridge-otel` + `opentelemetry-exporter-otlp`는 **HTTP/protobuf 방식**으로 내보내기 때문에 gRPC 포트에 연결하면 프로토콜 불일치 오류 발생.
- **해결**: 14개 서비스 `application.yml` 전부 `http://localhost:4318/v1/traces`(HTTP 포트)로 수정. Tempo 컨테이너는 4317(gRPC)·4318(HTTP) 둘 다 listen하므로 인프라 변경 불필요.
- **배운 점**: Spring Boot 3.3 OTLP exporter는 HTTP/protobuf 기본. gRPC exporter를 원하면 `opentelemetry-exporter-otlp` 대신 `opentelemetry-exporter-otlp-grpc` 의존성이 필요. 포트 4317(gRPC) vs 4318(HTTP) 구분 필수.

---

### [2026-06-28] WebFlux/Coroutine 서비스에서 traceId가 MDC에 전파되지 않음

- **증상**: api-gateway(WebFlux), market-collector(Coroutine) 로그에 `traceId`가 비어 있음(`[,,]`). servlet 기반 서비스는 정상.
- **원인**: Reactor와 Kotlin Coroutine은 ThreadLocal 기반 MDC를 지원하지 않음. Micrometer Tracing의 기본 컨텍스트 전파가 서블릿 ThreadLocal 방식으로만 동작.
- **해결**: 
  1. WebFlux 서비스(`api-gateway`, `realtime-gateway`): `Application.kt`에 `Hooks.enableAutomaticContextPropagation()` 추가.
  2. Coroutine 서비스(`market-collector`): `KisWebSocketClient` CoroutineScope에 `MDCContext()` 추가 + `Application.kt`에도 `Hooks.enableAutomaticContextPropagation()`.
- **배운 점**: Reactive/Coroutine 컨텍스트 전파는 별도 설정 필수. 서블릿 서비스에서 동작한다고 Reactive에서도 자동 동작하지 않음. `micrometer-context-propagation` 의존성도 추가 필요.
