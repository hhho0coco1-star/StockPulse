# StockPulse 기술 선택 이유

> 각 기술을 **왜 골랐는지 + 대안 + 트레이드오프**를 정리. 면접·리뷰에서 가장 자주 받는 질문에 대비한 근거 문서. (외부 API 실접근 검증은 Phase 0에서 수행)

---

## 1. 아키텍처 결정

### MSA (마이크로서비스)
- **선택 이유**: 수집(고빈도 스트리밍)·인사이트(CPU 집약)·실시간(WS 다중 연결)·모의투자(트랜잭션)는 **확장 특성이 완전히 다르다.** 서비스를 나눠 각자 독립 확장(예: Insight·Realtime만 Replica↑)하기 위함.
- **대안**: 모놀리식 → 초기 개발은 빠르나, 트래픽 특성이 다른 모듈을 함께 확장해야 해 비효율. 본 프로젝트 목표(고가용성·확장성 증명)와 안 맞음.
- **트레이드오프**: 운영 복잡도↑(13개 서비스), 분산 트랜잭션 필요 → 이를 Saga로 해결(아래).

### 이벤트 기반 (Kafka)
- **선택 이유**: 시세→인사이트→푸시의 **단방향 데이터 흐름**과 서비스 간 느슨한 결합. 한 서비스가 죽어도 이벤트가 큐에 남아 복구 후 재처리 가능.
- **대안**: 서비스 간 동기 REST 호출 → 장애 전파·강결합. 실시간 대량 흐름에 부적합.

### Saga (오케스트레이션) + Outbox
- **선택 이유**: 모의투자 주문이 Trading·Account·Portfolio 3개 서비스 DB를 바꾼다. 분산 환경엔 2PC가 비현실적 → **Saga로 단계별 처리 + 보상**. Trading이 상태 머신을 쥔 **오케스트레이션**을 택해 흐름 추적·디버깅을 쉽게.
- **Outbox 이유**: "DB 커밋"과 "Kafka 발행"의 원자성 문제(둘 중 하나만 성공) 방지. 비즈니스 변경과 이벤트를 같은 트랜잭션에 기록 후 발행 → **이벤트 무유실**.
- **대안**: 코레오그래피 Saga(이벤트만으로 연쇄) → 결합은 더 느슨하나 흐름 파악·보상 추적이 어려움. 학습·시연 목적엔 오케스트레이션이 유리.

## 2. 추가 스택 (SafeAlert 차별 — 왜 더했나)

| 기술 | 선택 이유 | 대안 / 트레이드오프 |
|------|-----------|---------------------|
| **Kafka Streams** | 틱 스트림 **윈도우 집계**(이동평균·RSI·거래량)로 모멘텀 지표 계산. 상태 기반 스트림 처리의 정석 | 대안 Flink(과한 운영부담)·직접 구현(상태·윈도우 재발명). Streams는 Kafka에 내장돼 추가 인프라 없음 |
| **TimescaleDB** | 틱·캔들은 시계열. 시간 파티셔닝·연속 집계로 차트 조회 최적화 | 대안 InfluxDB(별도 쿼리언어 학습)·일반 PG(대량 시계열 성능↓). Timescale은 PG 확장이라 SQL 그대로 |
| **Kotlin + Coroutines** | 다종목 시세/뉴스 **동시 수집**을 적은 스레드로. null 안전·간결성 | 대안 Java 스레드/CompletableFuture(장황). 트레이드오프: 팀 Kotlin 숙련 필요 |
| **Testcontainers** | 실제 Kafka·Redis·PG를 띄워 **Saga 보상**까지 통합 검증. mock보다 신뢰도↑ | 대안 임베디드/mock(실제와 괴리). 트레이드오프: 테스트 실행 시간↑ |
| **GitHub Actions + Helm** | 13개 서비스 빌드·테스트·배포 자동화. 선언적 패키징 | 대안 수동 kubectl(SafeAlert 방식, 재현성↓)·Jenkins(별도 서버) |
| **TradingView Lightweight Charts** | 캔들·실시간 차트의 사실상 표준 라이브러리 | 대안 직접 차트 구현(공수 큼)·Chart.js(금융 차트 약함) |
| **Loki + Tempo** | Grafana 하나로 메트릭+로그+트레이스 통합. ELK보다 경량·저비용 | 대안 ELK(무겁고 SafeAlert서 이미 함)·Jaeger(트레이스 전용). 트레이드오프: 풀텍스트 검색은 ELK가 강함 |
| **Chaos Mesh** | "장애 경험/해결" 스킬을 **체계적 장애 주입**으로 증명 | 대안 수동 장애 주입(재현성↓). 트레이드오프: K8s 환경 필요 |

## 3. 기존 스택 (SafeAlert에서 이어가는 것)

| 기술 | 선택 이유 |
|------|-----------|
| **Spring Boot / Cloud Gateway** | 검증된 생태계, Gateway 중앙 인증(JWT 검증 후 `X-User-Id` 전달)로 서비스 코드 단순화 |
| **Kafka** | 고처리량 이벤트 백본, 파티션 기반 순서 보장·소비자 그룹 확장 |
| **Redis** | 시세 캐시(O(1))·랭킹(Sorted Set)·Rate Limit(TTL)·**Pub/Sub fan-out**·세션 — 한 미들웨어로 다목적 |
| **PostgreSQL** | 트랜잭션·정합성이 중요한 주문·잔고·원장에 적합 |
| **MongoDB** | 뉴스·인사이트 같은 **스키마 유연한 문서**에 적합 |
| **WebSocket(STOMP)** | 시세·채팅·인사이트의 서버 푸시. Redis Pub/Sub과 결합해 다중 인스턴스 확장 |
| **Resilience4j** | 외부 API(KIS·네이버·DART) 장애 격리 — CB·retry·timeout·bulkhead |

## 4. 제외한 기술 (왜 안 썼나)

| 기술 | 제외 이유 |
|------|-----------|
| **Service Mesh (Istio)** | 앱 계층 Resilience4j로 retry/timeout/CB 이미 충족. mesh는 K8s 복잡도를 크게 올려 투자 대비 효과 낮음 |
| **WebFlux (리액티브)** | Coroutines로 비동기 충족. 풀 리액티브는 학습·디버깅 난도가 높아 일정 리스크 큼. 틱 백프레셔는 Kafka 소비 페이싱으로 제어 |
| **Apache Flink** | Kafka Streams로 윈도우 집계 충분. Flink는 별도 클러스터 운영 부담 |
| **GraphQL** | 화면별 조회는 REST로 충분. 스키마·캐싱 복잡도 대비 이득 적음 |
| **Keycloak** | 자체 JWT/OAuth2(SafeAlert 자산)로 충분. 별도 IdP 운영 불필요 |
| **Terraform** | 단일 클러스터 규모엔 과함. 필요 시 Phase 8에서 옵션 |

> 원칙: "있어 보이는 기술"이 아니라 **이 도메인 문제를 푸는 데 필요한 기술**만 선택. 차별성은 기술 나열이 아니라 소수 영역의 깊이에서.
